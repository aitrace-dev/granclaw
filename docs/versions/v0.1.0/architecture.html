agent-brother

v0.1.0 — MVP

What is this?

agent-brother is a minimal multi-agent AI framework. You run it locally, open a dashboard, and chat with AI agents that can use tools — browse the web, read/write files, run code. Every action is logged to MongoDB. The design is deliberately small so the entire codebase is readable.

It takes inspiration from three open-source projects: NanoClaw (small codebase, container isolation), IronClaw (security model, endpoint allowlisting), and ClaudeClaw (Claude Code CLI integration). The "Big Brother" oversight agent — which intercepts and can cancel any agent action in real time — is the core original idea.

v0.1.0 scope: one agent, no guardrails, working chat + logs dashboard. This version proves the core loop. Big Brother, multi-agent, and Docker isolation come in later phases.

How to run

# 1. Start MongoDB
docker compose up -d

# 2. Copy env
cp .env.example .env

# 3. Install dependencies (once)
npm install

# 4. Start everything
npm run dev
# → backend on http://localhost:3001
# → frontend on http://localhost:5173

Prerequisites: Node 20+, Docker, and the Claude Code CLI installed (claude must be on your PATH).

Big picture

There are three separate processes running at the same time:

┌─────────────────────────────────────────────────────────┐ │ YOUR MACHINE │ │ │ │ ┌──────────────┐ WebSocket ┌──────────────────┐ │ │ │ Browser │◄───────────────►│ Backend (3001) │ │ │ │ (port 5173) │ REST (HTTP) │ Express + WS │ │ │ └──────────────┘◄───────────────►└────────┬─────────┘ │ │ │ │ │ spawn │ child_process│ │ ▼ │ │ ┌────────────────┐ │ │ │ claude CLI │ │ │ │ (subprocess) │ │ │ └────────────────┘ │ │ │ │ │ reads │ │ │ ▼ │ │ ┌────────────────┐ │ │ │ MongoDB │ │ │ │ (port 27017) │ │ │ └────────────────┘ │ └─────────────────────────────────────────────────────────┘

The frontend (React) never talks to Claude directly — everything goes through the backend. The backend is the only piece that knows about Claude, MongoDB, and your config file. The claude CLI is a child process spawned per message — it runs, responds, and exits.

Config as code

Every agent is defined in agents.config.json at the project root. There is no database table for agent configuration — the file is the source of truth. The backend reads it fresh on every request, so changes take effect immediately without a restart. 

// agents.config.json
{
  "agents": [
    {
      "id":           "main-agent",        // unique, used in URLs + logs
      "name":         "main-agent",        // display name in UI
      "model":        "claude-sonnet-4-5", // passed to claude CLI
      "workspaceDir": "./workspaces/main-agent", // agent's working directory
      "allowedTools": ["filesystem", "browser"],
      "bigBrother":   { "enabled": false }  // P2: BB oversight
    }
  ]
}

In later phases, when you tell the admin agent "disable browser for main-agent", it will write to this file and commit it to git. The change is visible in version history — that's what "config as code" means.

Backend

The backend is a single Node.js + TypeScript process (packages/backend). It does four things:

index.tsEntry point. Connects to MongoDB, then starts the HTTP + WebSocket server.
server.tsExpress for REST routes. WebSocket server for real-time streaming. Both run on port 3001.
config.tsReads and parses agents.config.json. Used by routes and agent-runner.
db.tsMongoose connection + two schemas: AgentAction and AgentSession. Helper functions for logging and session lookup.

routes/logs.tsGET /logs — paginated log query with agentId/type filters.

Agent runner — the core

agent-runner.ts is where the AI runs. It uses the same technique as ClaudeClaw: instead of calling an API directly, it spawns the claude CLI as a child process and communicates with it via stdin/stdout.

Why the CLI instead of the API?

The Claude Code CLI handles tool use (filesystem, browser, bash) natively. You don't have to implement tool execution yourself — Claude calls tools, gets results, and reasons about them, all within the same process. If you called the raw Anthropic API, you'd have to implement every tool yourself.

What command gets run?

# First message (no session yet)
claude -p "your message" --output-format stream-json --verbose

# Follow-up messages (resume same conversation)
claude -p "your message" --output-format stream-json --verbose --resume abc123

The --output-format stream-json flag makes Claude emit one JSON object per line instead of a human-readable text block. Each line describes something that happened: a piece of text, a tool call, a tool result, or the final summary with the session ID.

What does the streamed output look like?

// Text chunk from Claude
{"type":"assistant","message":{"content":[{"type":"text","text":"I found 3 files..."}]}}

// Tool use (Claude calling a tool)
{"type":"assistant","message":{"content":[{"type":"tool_use","name":"filesystem","input":{...}}]}}

// Final result (session ID for memory)
{"type":"result","session_id":"abc123xyz","result":"Done."}

The runner parses each line, converts it to a StreamChunk (our own simpler type), and immediately forwards it to the WebSocket client. The user sees text appearing in real time, and tool calls appear as they happen.

WebSocket protocol

The frontend keeps a single persistent WebSocket connection open to the backend (port 3001). All real-time communication goes through it. The protocol is simple JSON messages.

Client → Server

{ "type": "message", "agentId": "main-agent", "text": "list files in /workspace" }

Server → Client (streamed, one per chunk)

// A piece of text
{ "type": "chunk", "agentId": "main-agent", "chunk": { "type": "text", "text": "I found..." } }

// A tool call
{ "type": "chunk", "agentId": "main-agent", "chunk": { "type": "tool_call", "tool": "filesystem", "input": {...} } }

// Conversation finished (includes new session ID)
{ "type": "chunk", "agentId": "main-agent", "chunk": { "type": "done", "sessionId": "abc123" } }

// Error
{ "type": "error", "agentId": "main-agent", "message": "claude exited with code 1" }

The frontend hook (useAgentSocket.ts) keeps a map of agentId → callback function. When a chunk arrives for "main-agent", it calls the callback registered by the ChatPage for that agent. This means in the future, multiple agents can stream simultaneously through the same connection.

MongoDB — memory and logging

MongoDB stores two things: every action an agent takes, and the session ID that lets conversations continue.

AgentAction collection — the audit trail

Every message in, every tool call, every tool result, every error, every system event is a document here.

FieldTypeWhat it holds
agentIdstringWhich agent this belongs to ("main-agent")
typeenummessage / tool_call / tool_result / error / system
inputMixedWhat went in (the user's text, the tool's arguments)
outputMixedWhat came back (tool result, exit code)
durationMsnumberHow long it took (set on system events at end of run)
createdAtDateAuto-set by Mongoose timestamps

AgentSession collection — conversation memory

The Claude CLI keeps full conversation history internally, indexed by a session ID. When a run finishes, Claude emits that session ID in the final JSON line. We save it to MongoDB keyed by agentId.

On the next message, the runner looks up the latest session ID for that agent, and passes it as --resume <sessionId> to the CLI. Claude reloads the conversation — the agent remembers everything from before.

What happens if you restart the backend? No problem — sessions are in MongoDB, not in memory. The next message will resume exactly where you left off.

FieldWhat it holds
agentIdWhich agent owns this session
sessionIdThe Claude session identifier string
lastActiveAtUpdated on every message — useful for "last seen" UI

UI overview

The frontend is a React single-page app built with Vite. It uses Tailwind CSS with custom design tokens from the "The Panopticon Aesthetic" design system (created in Stitch during the design phase). There are three pages.

All pages share the same shell: a 60px icon sidebar on the left and a top bar with the app name. The sidebar has four navigation items: Dashboard, Agents, Logs, Settings. The active item gets a green left-border indicator.

Dashboard screen /dashboard

Fetches GET /agents on load and renders one AgentCard per agent. Each card shows: name, model (in monospace green), a status badge (RUNNING / IDLE), which tools are enabled, and two action buttons — Chat and Logs.

The status comes from whether MongoDB has a session for that agent. If a session exists, it shows RUNNING (the agent has had at least one conversation). If not, IDLE. In a future version this will reflect whether the claude subprocess is actively running right now.

Chat screen /agents/:id/chat

This is the main working screen. It has two panels side by side:

Left panel (agent info): Fetches GET /agents/:id and shows the agent's name, model, status badge, and a list of its allowed MCP tools. Each tool appears with a green dot — these are read-only in v0.1.0 (toggles come in P2 when Big Brother can enforce them).

Right panel (chat): The actual conversation. Messages are stored in React state (not in the DB — only the backend logs to DB). When you type and press Enter or click Send:

  • 1

    Your message appears immediately in the chat as a user bubble

  • 2

    An empty agent bubble appears with a pulsing animation (streaming indicator)

  • 3

    The WebSocket hook sends { type: "message", agentId, text } to the backend

  • 4

    As chunks arrive, the agent bubble fills in with text in real time

  • 5

    Tool calls appear above the response as small monospace badges (e.g. ⚙ filesystem({path: '/workspace'}))

  • 6

    When the done chunk arrives, the pulsing stops and the input unlocks

Logs screen /logs

Fetches GET /logs and shows a compact table of everything MongoDB has recorded. Columns: Timestamp (ms precision), Agent, Type, Details (summarised from input/output), Duration (ms).

Five filter chips at the top let you narrow by type: ALL, MESSAGE, TOOL CALL, TOOL RESULT, ERROR. If you arrive from an agent card's "Logs" button, the URL is pre-filtered to that agent's ID. Rows are colour-tinted: error rows get a faint red background.

Full message lifecycle

Here is exactly what happens from the moment you press Enter to when you see the reply:

USER presses Enter │ ▼ ChatPage.tsx (React) - adds user bubble + empty agent bubble to local state - calls useAgentSocket.sendMessage(agentId, text, onChunk) │ ▼ WebSocket frame server.ts (Express/WS) - receives { type:"message", agentId:"main-agent", text:"..." } - looks up agent in agents.config.json via config.ts - calls runAgent(agent, text, onChunk) │ ▼ agent-runner.ts - getSession(agentId) → MongoDB lookup → returns sessionId or null - logAction(agentId, "message", { text }) → writes to MongoDB - spawns: claude -p "..." --output-format stream-json [--resume id] │ ▼ stdout (line by line) claude CLI (subprocess) - reasons about the message - calls tools (filesystem, browser, etc.) and gets results - emits JSON lines as it goes │ ▼ each line parsed by agent-runner agent-runner.ts (stream parser) - converts claude JSON events → StreamChunk objects - calls onChunk(chunk) for each one │ ▼ onChunk callback server.ts - ws.send(JSON.stringify({ type:"chunk", agentId, chunk })) │ ▼ WebSocket frame useAgentSocket.ts (React hook) - routes chunk to the ChatPage's handler by agentId │ ▼ ChatPage.tsx - appends text to agent bubble (type: "text") - adds tool badge (type: "tool_call") - unlocks input (type: "done") │ ▼ after "done" chunk agent-runner.ts - saveSession(agentId, newSessionId) → MongoDB upsert - logAction(agentId, "system", null, { exitCode }, durationMs)

Session memory — how the agent remembers

The Claude CLI manages conversation history internally. When it finishes a run it emits a session_id string. That ID is a pointer to the conversation stored on disk by the CLI.

Message 1: "What files are in /workspace?" agent-runner → no session in DB → runs claude without --resume claude creates new session → emits session_id: "ses_abc123" agent-runner → saveSession("main-agent", "ses_abc123") Message 2: "Now delete the temp ones" agent-runner → getSession("main-agent") → "ses_abc123" runs: claude -p "..." --resume ses_abc123 claude reloads history → already knows about the files from message 1 agent acts on the full context

Memory survives restarts. Session IDs live in MongoDB, not in process memory. Kill and restart the backend — the next message resumes the same conversation.

Memory is per-agent, not per-user. In v0.1.0 there is one session per agent. Multi-user or multi-session support is a future concern.
